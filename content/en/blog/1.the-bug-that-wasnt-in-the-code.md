I'm building a real-time multiplayer mode for a hanafuda card game. The game state gets serialized, encrypted, hashed for integrity, and synced through Firestore. When I tested between two browsers on localhost, everything worked. Two deployed clients, same thing. But testing between localhost and the deployed preview branch, the joining player got a blank table. No cards dealt, no error in the UI. Just an empty board.

The console had more to say: "Save data integrity verification failed — data may have been tampered with." So I started digging.

The integrity hash uses FNV-1a over the encrypted card data, a base salt from an environment variable, and an associated data salt derived from cross-store state. I spent a while convinced the associated salt components were drifting between environments. I checked all eight values that go into it: game ID, round counter, turn counter, turn phase, active player, bonus multiplier, max rounds, viewings yaku setting. All matched.

Then I looked at the base salt. The app reads `NUXT_PUBLIC_SAVE_INTEGRITY_SALT` from runtime config. I checked Cloudflare Pages. The variable was set. So that should be fine, right?

Here's the thing: this is a Nuxt SPA with SSR disabled. Public runtime config gets baked into the client bundle at build time. It doesn't matter what the environment variable is set to now; what matters is what it was when the build ran. I pulled the deployed bundle and found `saveIntegritySalt:""`. The variable was configured for Production but not for Preview. An empty string is falsy in JavaScript, so the code fell through to a hardcoded fallback. Localhost had the real salt from `.env`. Different salts, different hashes, failed verification every time.

Here's what I learned.

When your app bakes config at build time, "the env var is set" and "the env var was set when this artifact was built" are two different things. SPA frameworks are sneaky here because there's no server to override values at runtime. The value is whatever `process.env` returned during `nuxt build`, frozen into a JavaScript file, served from a CDN.

The other thing: the game showed zero feedback when deserialization failed. It caught the error, logged it to console, and kept going with empty stores. I added toast notifications at every sync failure point so the next person to hit this sees "Failed to sync game state" instead of staring at a blank board. That's the kind of thing you don't think about until you're the one staring at it.
